Archive Detective · A-to-Z handoff

Turn the newspaper demo into a mixed-evidence archive mystery game.

The current project proves the pipeline, but newspaper-only gameplay feels too much like OCR dressed up as a game. The next agent should pivot the product into Archive Detective: Evidence Cabinet: a local-first detective board where each case is assembled from public-domain artifacts: maps, letters, photos, court records, objects, postcards, and newspapers.

1. Product North Star

One-sentence pitch

Open a public-domain archival case box, inspect real artifacts, and reconstruct what happened from clues the model helps organize.

What it is not

It is not a generic archive search app, not a chatbot, and not “newspaper OCR in a nice UI.” Newspapers become one artifact type inside a richer case file.

Design scene: A late-night archivist opens a wooden evidence cabinet under a desk lamp. The player pulls physical-feeling records from trays, compares them, and writes a final deduction. The UI should feel tactile and purposeful, not like a dashboard.

Primary goals

  • Make the product instantly more fun by giving players multiple artifact types per case.
  • Preserve the local-first path: all demo assets live on disk, no cloud API required at play time.
  • Keep `gradio.Server` as the custom backend/UI foundation for the Off-Brand badge.
  • Keep llama.cpp optional but wired as the route for local generated clue cards.
  • Do not chase fine-tuning unless everything else is already excellent.

2. Source Strategy: Better Than Newspapers

The next agent should use mixed public-domain evidence sources. The goal is not to build every integration immediately. The goal is to support the data shape and create one polished hero case that proves the new format.

Source Why It Helps Access Notes Priority
Library of Congress Maps, manuscripts, photos, books, newspapers, IIIF images, no key. Use loc.gov JSON API and IIIF images. Formats include `maps`, `photos`, `manuscripts`, `newspapers`. Hero source
NYPL Digital Collections Public-domain photos, maps, manuscripts, prints, ephemera. API supports `publicDomainOnly=true`, but may need API token. Good later integration. Next
Smithsonian Open Access CC0 objects, portraits, tools, artifacts, 2D/3D images. API key needed for live API; bulk metadata exists. Great object evidence. Next
Old Bailey Online Criminal trial transcripts, 1674–1913. Strong mystery material. API/XML. Not U.S. public-domain archive, but very relevant to deduction gameplay. Excellent case source
Dallas Police Historical Case Files Actual police case files, photos, letters, clippings, 1920s–1950s. Portal to Texas History collection. Rights must be checked per item. High impact
DPLA Broad discovery layer across libraries and museums. Metadata only, API key required, links out to originals. Good for discovery, not first build. Later

Recommended first hero case

The Hart Notice, expanded

Use the existing Georgetown notice as the first artifact, then add 3–4 fabricated-but-clearly-marked local fixture artifacts only if public-domain assets cannot be found fast. Prefer real LOC map/photo assets if available.

The Midnight Brief, expanded

Use the 1937 “organ signal” newspaper clipping as a sensational anchor. Add venue photo, Los Angeles map crop, temple-related public-domain image, and a short directory/card artifact.

If time is tight, expand one existing case manually with local static artifacts and citations. Build the game layer first, then add more ingest automation.

3. Gameplay Loop

Core mechanics to implement

MechanicPurposeFirst implementation
Artifact tray Makes cases feel like physical evidence boxes. Horizontal list of artifact cards with type icons and status.
Artifact viewer Focus on one artifact at a time: image/text/metadata. Replace single clipping panel with selected artifact panel.
Evidence pins Give players digestible clue cards extracted from artifacts. Show cards grouped by artifact and allow “pin to board”.
Search console Her Story-style active play, not passive reading. Local search across artifact OCR/transcripts; highlight matching artifacts.

4. Target Architecture

Architectural decisions

  • Keep `gradio.Server`, not `gr.Blocks`, for the custom UI badge and better UX freedom.
  • Keep old `CluePack` compatibility during migration so current cases still run.
  • Add `EvidenceCase` and `Artifact` models rather than breaking every existing file at once.
  • Make ingest scripts dev-only. Runtime should load local JSON/assets and optionally local llama.cpp.
  • Do not deploy to Hugging Face until user explicitly says to deploy.

5. Data Model Plan

Current model is one `CluePack` per beat with a single `Fragment`. The new product needs cases containing multiple artifacts. Do this with additive models first, then migrate UI and engine.

Keep current models

  • `Source`
  • `Fragment`
  • `Entity`
  • `EvidenceCard`
  • `LeadOption`
  • `RevealNotes`
  • `CluePack`
  • `CaseDefinition`

Add new models

  • `ArtifactKind` literal/string
  • `Artifact`
  • `DeductionField`
  • `DeductionSheet`
  • `EvidenceCase`
  • `CaseManifest` optional

Compatibility rule

`load_case()` should support both old `CaseDefinition` and new `EvidenceCase`. If `mode` is missing, treat it as the old beat-based case. If `mode == "evidence_cabinet"`, use the new path.

6. UI/UX Plan

The UI should move from “evidence dashboard” to “case box.” Keep the dark archive desk, but reduce generic panels and make interactions feel physical.

Left: Evidence Tray

Artifact cards with thumbnails, kind labels, locked/unlocked state, and citation hint.

High fun

Center: Artifact Viewer

Large selected artifact. Image, OCR, metadata, hotspots, search highlights.

Core play

Right: Deduction Sheet

Clue pins, leads, final fill-in fields, reveal facts.

Payoff

Specific UI changes

  • Change header copy from “newspaper fragments” to “real archive evidence.”
  • Replace single `Exhibit A` with selected artifact viewer.
  • Add artifact tray section to `index.html` and render via `board.js`.
  • Use type-specific labels: Map, Letter, Photo, Object, Trial, Notice, Clipping.
  • Add local text search input: search all artifact text, show matching count and highlight selected artifact.
  • Add deduction sheet UI. First version can be multiple-choice/select fields, not freeform NLP.
  • Add “Archive facts” reveal as the final state after deduction attempt.
  • Use real citations in a compact citation drawer per artifact.
  • Browser test at desktop and narrow width.

Design rules for the next agent

  • No default Gradio components visible in the main UI.
  • No generic dashboard metrics, no SaaS cards, no fake “AI magic” glow.
  • Make artifacts feel tactile: trays, labels, tabs, sleeves, stamps, magnifier affordance.
  • Keep copy direct: “Open artifact,” “Pin clue,” “Search evidence,” “Submit deduction.”
  • Use motion sparingly: artifact selection, reveal drawer, search result highlight. Respect reduced motion.

Update case loading and API serialization to return either old beat state or new evidence-case state.

Build `data/cases/hart_notice_evidence_cabinet.json` or `midnight_brief_evidence_cabinet.json` manually first.

  • Minimum 4 artifacts: newspaper, map/photo, letter/directory-style text, object/photo/card.
  • Every artifact has a citation or clear synthetic/test flag.
  • Minimum 6 evidence cards across artifacts.
  • Minimum 3 leads that unlock or focus artifacts.
  • One deduction sheet with 3 fields.

Phase 4: Rework board UI into evidence cabinet

Update static HTML/CSS/JS. Do not return to `gr.Blocks`.

  • Add left evidence tray.
  • Add center artifact viewer.
  • Add right deduction sheet and clue pins.

Phase 5: Local llama.cpp integration polish

Keep llama optional. The app must run without it. Use it to regenerate clue cards locally when available.

  • Confirm `scripts/run_llama.sh` works when GGUF exists.
  • Add `ARCHIVE_DETECTIVE_USE_LLAMA=1` docs.
  • Use llama for text artifacts first, not image reasoning.
  • Fail gracefully if llama server is not running.
